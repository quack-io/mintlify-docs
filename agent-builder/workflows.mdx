Workflows let you define orchestrated sequences of tool executions. Instead of relying on the LLM to chain tools together, a workflow specifies the exact order, conditions, and branching logic — giving you deterministic multi-step processes.

## When to use workflows

- **Multi-step operations** — verify identity, look up account, apply discount, send confirmation
- **Conditional branching** — take different paths based on tool outputs or parameter values
- **Parallel execution** — run independent tools simultaneously for faster resolution
- **Retry and fallback logic** — automatically retry failed tool calls or fall back to alternatives

<Info>
  Workflows execute tools in a defined order. Rules still apply within each step — if a rule matches during a workflow step, it takes priority.
</Info>

## Workflow object

<ResponseField name="id" type="string">
  Unique identifier (e.g., `wf_01J...`).
</ResponseField>

<ResponseField name="name" type="string">
  Display name.
</ResponseField>

<ResponseField name="description" type="string">
  What this workflow accomplishes.
</ResponseField>

<ResponseField name="scope" type="string">
  Visibility scope: `organization`, `account`, or `network`.
</ResponseField>

<ResponseField name="tools" type="object[]">
  Ordered list of tool references with execution configuration.
  <Expandable title="tool step fields">
    <ResponseField name="tool_id" type="string">
      Reference to a configured tool.
    </ResponseField>
    <ResponseField name="step" type="integer">
      Execution order (1-based).
    </ResponseField>
    <ResponseField name="condition" type="object">
      Optional condition group (`all`/`any`/`not`) — step only executes if the condition matches.
    </ResponseField>
    <ResponseField name="on_failure" type="string">
      Behavior when the tool call fails: `stop`, `skip`, or `retry`.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="parameters" type="string[]">
  Parameter codes required by this workflow.
</ResponseField>

<ResponseField name="strategies" type="object">
  Execution strategies.
  <Expandable title="strategy fields">
    <ResponseField name="execution_mode" type="string">
      `sequential` (default) or `parallel`.
    </ResponseField>
    <ResponseField name="max_retries" type="integer">
      Maximum retry attempts for failed steps.
    </ResponseField>
    <ResponseField name="timeout_ms" type="integer">
      Maximum execution time in milliseconds.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="custom_settings" type="object">
  Arbitrary key-value settings for workflow-specific configuration.
</ResponseField>

<ResponseField name="feature_flags" type="object">
  Feature flags to enable or disable experimental workflow behavior.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp.
</ResponseField>

<ResponseField name="updated_at" type="string">
  ISO 8601 timestamp.
</ResponseField>

```json Example
{
  "id": "wf_01J...",
  "name": "Process Return",
  "description": "Verify order, check eligibility, initiate return, and send confirmation",
  "scope": "organization",
  "tools": [
    {
      "tool_id": "tool_01J_lookup",
      "step": 1,
      "on_failure": "stop"
    },
    {
      "tool_id": "tool_01J_eligibility",
      "step": 2,
      "condition": {
        "all": [
          { "parameter": "order-status", "operator": "equals", "value": "delivered" }
        ]
      },
      "on_failure": "stop"
    },
    {
      "tool_id": "tool_01J_initiate_return",
      "step": 3,
      "on_failure": "retry"
    },
    {
      "tool_id": "tool_01J_send_email",
      "step": 4,
      "on_failure": "skip"
    }
  ],
  "parameters": ["order-id", "order-status", "return-reason"],
  "strategies": {
    "execution_mode": "sequential",
    "max_retries": 2,
    "timeout_ms": 30000
  },
  "custom_settings": {},
  "feature_flags": {},
  "created_at": "2026-03-05T12:00:00Z",
  "updated_at": "2026-03-09T10:00:00Z"
}
```
