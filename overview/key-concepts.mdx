Every Apollo-1 agent has two sides.

- A **frontend** that faces users — it converses, it acts, and when business logic has been defined, it enforces absolutely.
- A **backend** that faces the entity — where you teach the agent how to act, ask it why it did something, and evolve its behavior.

Both sides read and write the same symbolic substrate. Enforcement and editing happen against one set of files, in one vocabulary, with one trace.

## Agent Frontend: Formal Enforcement

Apollo-1 ships with a Playground where any use case runs from the system prompt alone. When you define your tools, Apollo-1 automatically generates an **ontology**: a structured representation of your entities, parameters, and relationships. This ontology is shared across all your tools. Business logic defined once applies everywhere it's relevant.

From the ontology, you define business logic for the scenarios where the agent must reason formally. Apollo-1 supports a growing set of constraint types, each enforced symbolically at runtime.

### Policy Constraints

Business rules the agent must enforce unconditionally. These are the hardest guarantees — actions that must be blocked or required regardless of conversational context.

> "Block disputes for transactions older than 8 days."
> "Never process a plan downgrade during an active billing dispute."
> "Do not schedule appointments outside business hours."

### Confirmation Constraints

Actions that require explicit user consent before execution. The agent must pause, present the action it's about to take, and wait for affirmative confirmation.

> "Require confirmation before processing payment."
> "Show the user the full cost breakdown and get explicit approval before booking."

### Authentication Constraints

Actions that require identity verification before execution.

> "Require ID verification for refunds over $200."
> "Verify account ownership before changing billing information."

### Conditional Constraints

Rules that apply only when specific conditions are met — combining state evaluation with policy enforcement to handle branching logic that would normally require hand-coded workflows.

> "Allow same-day cancellation only for Business Class passengers with Platinum Elite status."
> "Escalate to a human agent if the claim amount exceeds $10,000 and the policy is less than 90 days old."

### Rate and Threshold Constraints

Limits on frequency, volume, or value that the agent must track across interactions and enforce cumulatively.

> "Limit each customer to three dispute submissions per billing cycle."
> "Cap courtesy credits at $50 per account per quarter."

### Sequencing Constraints

Rules that enforce ordering — actions that must happen before other actions, or that are only valid at specific points in a workflow.

> "Collect shipping address before presenting delivery options."
> "Run a credit check before presenting loan terms."

Express these in natural language; Apollo-1 compiles them into symbolic reasoning.

<Note>
These aren't instructions the agent tries to follow. They're logic — evaluated symbolically, enforced formally. Once state is correctly interpreted, the outcome is guaranteed.
</Note>

### How Enforcement Works

Logic enforcement is formal; perception is not. When the Symbolic Reasoning Engine evaluates a rule, the evaluation is deterministic. If the predicate says `today - txn.date <= 8 days` and the transaction is 9 days old, the action is blocked. Every time.

Perception — understanding what the user is asking — remains probabilistic, handled by the neural components. The system may misunderstand what's being requested. But it won't "decide" to skip a required step or "forget" a policy mid-conversation. Misclassification affects whether an action is attempted, not whether business logic is enforced. If perception fails, the action isn't invoked; the user experiences task failure, not policy violation.

## Agent Backend: Open Cognition

The agent backend is the side that faces the entity. In every other AI architecture, there's a hard wall between talking to the agent and changing the agent. Apollo-1 dissolves it, for a structural reason: the same property that lets the engine enforce a rule is the property that lets a human read it. The rule is a real symbolic object in a real file.

**Open cognition** is what we call this property of the architecture. The cognition is open in the way open-source code is open: it can be read, located, modified, diffed, and verified.

The Apollo-1 Playground exposes the agent backend as a second model with read-write access to the agent's symbolic structures. You talk to it in natural language. It can:

- **Explain decisions** by reference to the actual symbolic trace — not a post-hoc rationalization. *Why did you block that?* has a literal answer: this rule, this predicate, this state.
- **Locate rules** by description. You don't need to know the file or the schema, just the policy.
- **Compile intent into formal logic** and apply it as a diff against the agent's configuration.
- **Evaluate changes** against other scenarios and edge cases before pushing live.
- **Verify** that the change fires correctly in the next turn.

### Example: Credit Card Dispute Agent

Configured with the policy *"block disputes for transactions older than 8 days."*

**Frontend — Enforcement.** The user asks to dispute a charge from 9 days ago. The planner selects Create Dispute and populates the parameters. Before the call executes, the Symbolic Reasoning Engine evaluates the policy constraint. The predicate `today - txn.date <= 8 days` returns false. The block is logged; the tool call does not happen. The agent talks like an LLM and refuses like a compiler.

**Backend — Editing.** In the agent backend: *"Change the dispute blocking rule from 8 days to 3 days."* The backend locates the existing rule, produces a diff against `rules.aui.json`, updates the user-facing explanation to match the new threshold, and pushes the change live.

**Frontend — Verification.** In a fresh conversation, the agent refuses to dispute a charge from 4 days ago — disputable under the old rule, blocked under the new one. Same trace shape. New predicate, new threshold.

The loop takes under a minute. No engineer. No redeployment. Every step auditable.

## Why This Matters

Formal enforcement and open cognition are not two separate features. They are two consumers of the same substrate. When the agent's brain is addressable in natural language, anyone who can describe a policy can author it, change it, and verify it. Engineers stop being a bottleneck for business logic, because business logic stops being an engineering artifact.
